# Full infrastructure rebuild

This runbook is the entry point for recreating LabOps when the infrastructure is
lost. It records ownership, dependency order and non-recoverable prerequisites;
subsystem READMEs own their detailed commands.

## Reconstruction rules

- Git is the source of truth for declarative configuration.
- Terraform owns Cloudflare, Tailscale DNS, Proxmox networking and `labmgmt`.
- CAPI running on `labmgmt` owns `labtest` and `labprod`.
- Argo CD owns platform services and workloads after its one-time bootstrap.
- Sealed Secrets in Git contain ciphertext only. Their controller private keys
  require an encrypted backup outside Git if sealed values must survive cluster
  recreation without being resealed.
- Terraform state, kubeconfigs, Talos configs and plaintext credentials are not
  reconstruction documentation.

## Inputs that Git cannot recreate

| Input | Minimum scope | Safe source or recovery location | Consumer |
| --- | --- | --- | --- |
| Proxmox API credential | Only the privileges documented by the Proxmox and CAPI runbooks | Password manager or freshly issued token | Terraform and CAPMOX |
| Tailscale OAuth client | DNS write and the network scopes required by the network stack | Tailscale admin console; store only in an ignored variables file | `terraform/tailscale-dns` and networking |
| Cloudflare infrastructure token | Permissions documented by `terraform/cloudflare` | Cloudflare dashboard or password manager; ignored credential file | Cloudflare Terraform provider |
| Cloudflare cert-manager token | `Zone:DNS:Edit` and `Zone:Zone:Read`, restricted to `bingops.com` and `lab.bingo` | Cloudflare dashboard or password manager; seal independently for both clusters | cert-manager DNS-01 on labtest and labprod |
| Cloudflare tunnel credential | Generated by the Cloudflare Terraform stack | Read sensitive Terraform output directly into the sealing pipeline; never print it | cloudflared on `labprod` |
| GitHub access | Read access for public reconciliation; write access only for operators | GitHub account or deploy credential | Operator and Argo CD |
| Sealed Secrets recovery keys | Controller-generated private sealing keys | Encrypted offline backup, one set per cluster | Sealed Secrets recovery |

Record only where each secret can be recovered and how to rotate it. Never copy
its value, Terraform sensitive output, or an unsealed Kubernetes Secret here.

## Dependency order

1. Restore operator access and the external credentials listed above.
2. Recreate Proxmox networking and Tailscale routes using the owning Terraform
   and Ansible runbooks.
3. Recreate the Terraform-owned Talos management cluster `labmgmt`; verify both
   the Talos API and Kubernetes API before continuing.
4. Initialize CAPI providers and recreate the CAPI-owned workload clusters
   `labtest` and `labprod` according to [`capi/README.md`](../capi/README.md).
5. Rebuild local Kubernetes and Talos client configuration using
   [`hacks/README.md`](../hacks/README.md).
6. Confirm that `master` exists on the Git remote. Bootstrap Argo CD once on
   each workload cluster, then install the matching root Application described
   by [`apps/README.md`](../apps/README.md). Both roots follow `master`; their
   cluster-specific paths remain distinct.
7. Restore or reseal cluster-specific secrets only after the Sealed Secrets
   controller is healthy. Never reuse ciphertext with a different sealing key.
8. Apply split DNS for `*.test.lab.bingo` through `192.168.10.170` and for the
   exact private name `argocd.lab.bingo` through `192.168.10.160`. Confirm the
   Tailscale subnet route to `192.168.10.0/24`, then apply public Cloudflare
   DNS/tunnel configuration for `auth.lab.bingo`, `portfolio.lab.bingo`,
   `bingops.com` and `www.bingops.com`. Production `lab.bingo` routes are explicit; do not add
   either Argo CD hostname to the Cloudflare tunnel.
9. Recreate Authentik according to [`authentik.md`](authentik.md), restore its
   PostgreSQL identity backup, and verify both Argo CD OIDC clients. On a new
   installation, complete the documented one-time administrator bootstrap.
10. Push feature changes, integrate them temporarily into `labtest` with
   `hacks/deploy.sh`, validate the reconciled revision, restore labtest to its
   `master` baseline, then merge the reviewed feature into `master`.

## Recovery gates

Do not continue to the next layer until the current gate passes:

| Layer | Non-sensitive verification |
| --- | --- |
| Network | VLAN gateways and required Talos/Kubernetes ports are reachable |
| `labmgmt` | Node is Ready and both Talos and Kubernetes APIs answer |
| CAPI workloads | `labtest` and `labprod` report Available and their nodes are Ready |
| Argo CD | Controllers are Ready and Applications are Synced/Healthy |
| Secrets | Sealed Secrets controller is Ready; required generated Secrets exist without printing them |
| Test DNS | An arbitrary `<app>.test.lab.bingo` resolves privately to the labtest ingress address |
| Production DNS | `auth.lab.bingo`, `portfolio.lab.bingo`, `bingops.com`, and `www.bingops.com` resolve through Cloudflare |
| Private Argo CD DNS | `argocd.test.lab.bingo` resolves to `192.168.10.152` and `argocd.lab.bingo` resolves to `192.168.10.151` only for clients using Tailscale split DNS |
| Authentik | Both OIDC discovery documents return successfully over trusted HTTPS |
| TLS and routing | HTTPS validates with the normal public trust store on both clusters; labtest remains unreachable outside LAN/Tailscale |
| Argo CD access | Test rejects sources outside LAN/Tailscale; production remains unrouted publicly and both require successful Authentik login |

## Documentation completion criteria

A lifecycle change is incomplete if reconstruction still depends on remembered
console clicks, shell history, chat output, an unexplained identifier, or an
unrecorded credential scope. Document the prerequisite or refactor it away in
the same change, keeping detailed commands in the closest owning README.
